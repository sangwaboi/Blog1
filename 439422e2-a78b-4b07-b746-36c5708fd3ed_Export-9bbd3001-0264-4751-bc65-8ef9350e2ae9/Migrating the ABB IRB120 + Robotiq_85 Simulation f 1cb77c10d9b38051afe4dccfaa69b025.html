Migrating the ABB IRB120 + Robotiq_85 Simulation from ROS1 to ROS2 MoveIt2

When I first discovered the ABB IRB120 with the Robotiq_85 gripper simulation running on ROS1, I was immediately captivated by its sturdy design and vast potential. But as I dug deeper, I realized that working with ROS Kinetic, Gazebo 7, and rospy felt like navigating with outdated maps.. especially when dreaming of a dynamic, multi-robot future. That’s when I decided: it’s time for a major upgrade. Migrating to ROS2 Humble and MoveIt2 has opened up an entirely new universe of possibilities, transforming our simulation into a state-of-the-art platform that’s ready for real-time, collaborative industrial robotics.

The Orignal ROS1 Architecture :-

The original ROS1 system was built around:

• Gazebo simulation with ROS-Control: Using gazebo_ros_control to manage the interaction between the ABB IRB120 and the Robotiq gripper.

• XML-based launch files: Tightly integrated with roslaunch, all nodes (Gazebo, MoveIt, controllers, RViz) were started using static XML launch configurations.

• MoveIt1 motion planning stack: Configured with simple YAML files for kinematics and joint limits, with the planning server (move_group) handling motion planning.

• rospy-based Python scripts: For executing forward and inverse kinematics, trajectory planning, and control—using synchronous, single-threaded callbacks.

While robust, this setup was inherently limited by ROS1’s single-threaded execution model, global parameter server, and outdated middleware challenges that stifled multi-robot collaboration and real-time responsiveness.

______________________________________________________________________________

Motivation for Migration :-

The switch to ROS2 Humble and MoveIt2 was driven by transformative improvements:

• Multi-threaded execution: ROS2’s executors allow parallel callback handling, greatly enhancing responsiveness.

• DDS middleware: Native multi-robot support through decentralized discovery removes the need for fragile multi-master setups.

• Advanced MoveIt2 planners: Features like hybrid and reactive planning open the door to sophisticated, real-time motion planning.

These enhancements not only overcome ROS1’s limitations but also set the stage for innovative industrial exercises
such as collaborative multi-robot pick-and-place, vision-guided assembly, and reactive trajectory execution—that are at the heart of JdeRobot’s vision.

The Migration Process :-

1. Upgrading the Build System

This can be considered as a part of Group Query Attention where number of kv heads is 1. It just means that we are sharing query heads for a single head of key and values.

converted the ROS1 build system (catkin) to ROS2’s ament_cmake. This involved updating:

• package.xml: Now uses format 3 and specifies <build_type>ament_cmake</build_type>.

• CMakeLists.txt: Switched from find_package(catkin ...) to ROS2’s ament_cmake and individual find_package() calls for dependencies like rclcpp, rclpy, and std_msgs.

       <export>

2. Rewriting the Launch System

All ROS1 XML launch files were rewritten into sleek, Python-based ROS2 launch scripts. For example, the original irb120_robotiq85_gazebo.launch became a Python file that dynamically includes the Gazebo empty world launch, spawns the robot (using xacro to load the URDF), and starts the necessary controller spawners—all with improved modularity and parameter management.

Code Snippet from: irb120_robotiq85_gazebo_moveit_rviz.launch.py

#!/usr/bin/env python3
import os

from launch.actions import DeclareLaunchArgument, IncludeLaunchDescription, GroupAction

    robot_description = {'robot_description': robot_description_content}

    # ...

if __name__ == '__main__':

This new launch structure provides a modular, composable approach and improved node dependency management.

3. Updating Controller & MoveIt2 Configuration

Controller configurations were updated to work with ROS2’s ros2_control. The original YAML files were reused with minor modifications, ensuring that the URDF/Xacro and SRDF files remained intact. This preserved the original kinematic and semantic definitions while aligning the control interface with ROS2 standards.

Code Snippet: Example Controller YAML (ros_controllers.yaml)

controller_manager:
  ros__parameters:
    update_rate: 500
    joint_state_broadcaster:
      type: joint_state_broadcaster/JointStateBroadcaster
    arm_controller:
      type: joint_trajectory_controller/JointTrajectoryController
      joints: [joint_1, joint_2, joint_3, joint_4, joint_5, joint_6]
    gripper_controller:
      type: gripper_controller/GripperCommandController
      joints: [gripper_finger1_joint]

This YAML is now loaded directly as node parameters rather than via a global parameter server.

______________________________________________________________________________

4. Porting Python Nodes from rospy to rclpy

All custom Python scripts (e.g., forward kinematics, inverse kinematics, IK_cartesian_path) were manually ported from rospy to rclpy. This included replacing rospy.init_node() with rclpy.init() and using the node’s logger for output.

Code Snippet: ROS2 Python Node Example

import rclpy
from rclpy.node import Node

def main(args=None):
    rclpy.init(args=args)
    node = rclpy.create_node('example_node')
    node.get_logger().info('Hello from ROS2!')
    rclpy.shutdown()

if __name__ == '__main__':
    main()

5. Preserving the Robot Description

One of the most critical parts was retaining the original robot models. I kept the existing URDF/Xacro and SRDF files almost entirely intact, ensuring that the robot’s kinematic structure, visual fidelity, and semantic groups remained unchanged. This approach not only saved significant time but also guaranteed that the core functionality wasn’t compromised—allowing direct comparison between ROS1 and ROS2 behaviors.
